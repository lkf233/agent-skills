# 后端开发规范（MVC 版）

## 1. 代码规范

### 1.1 注释要求
- 禁止使用 Lombok 注解。
- 每个类、方法、字段都必须包含完整的 Javadoc 注释。
  - 注释需说明目的、参数、返回值及可能抛出的异常。
  - 目的：保证代码开源质量，便于教学和维护。

### 1.2 命名约定
- 方法命名：
  - `getXxx()`：必须返回非空结果，否则抛出异常。
  - `findXxx()`：允许返回 `null`。
  - `checkXxxExist()`：执行存在性检查并抛出异常。
  - `existXxx()`：返回布尔型存在性结果。
- 类命名：
  - API 入参：`XxxRequestObject`
  - API 出参：`XxxResponseObject`
  - 服务层对象：`XxxDTO`
  - 数据库实体：`XxxEntity`
  - 数据访问：`XxxMapper`
  - 业务服务：`XxxService`
  - 对象转换：`XxxAssembler`

## 2. 架构规范

### 2.1 分层架构（传统 MVC）
- **Controller 层（API 层）**
  - 使用 RESTful 路由风格。
  - 接收参数使用 `XxxRequestObject`。
  - 必须使用 `@Validated` 进行基础数据校验。
  - 负责请求接收、参数校验、响应封装，不承载业务逻辑。
- **Service 层（业务层）**
  - 负责业务流程编排和业务规则校验。
  - 使用 `XxxDTO` 在服务层内部传递数据。
  - 负责数据存在性检查、权限校验、事务边界控制。
- **Mapper 层（持久化层）**
  - 负责数据库访问和 SQL 映射。
  - 返回 `XxxEntity`，不承载业务逻辑。
  - 必须明确查询语义和更新结果。
- **UserContext 约束**
  - 当前登录用户信息统一通过 `UserContext` 获取，禁止在 Controller 中直接读取 `HttpServletRequest` 属性。
  - `UserContext` 由拦截器注入并在请求结束时清理，避免线程污染。

### 2.2 对象转换
- 使用 `XxxAssembler` 专用类处理对象转换。
- 允许合理使用 `BeanUtils.copy` 进行简单属性复制。
- 转换规则：
  - Controller -> Service：`RequestObject` -> `DTO`
  - Service -> Mapper：`DTO` -> `Entity`（按需要）
  - Mapper -> Service：`Entity` -> `DTO`
  - Service -> Controller：`DTO` -> `ResponseObject`

### 2.3 鉴权工具约束
- Token 的生成与解析统一通过工具类封装（例如 `JwtTokenUtil`），禁止在多个 Service/Controller 中重复实现。
- Controller 与 Interceptor 仅调用工具类，不直接拼装 JWT 细节。

## 3. 数据库操作规范

### 3.1 查询操作
- 必须明确区分严格查询和宽松查询：

```java
/**
 * 获取服务商信息（必须存在）
 * @param providerId 服务商ID
 * @param userId 用户ID
 * @return 服务商实体
 * @throws BusinessException 当服务商不存在时抛出
 */
public ProviderEntity getProvider(String providerId, String userId) {
    Wrapper<ProviderEntity> wrapper = Wrappers.<ProviderEntity>lambdaQuery()
        .eq(ProviderEntity::getId, providerId)
        .eq(ProviderEntity::getUserId, userId);
    ProviderEntity provider = providerMapper.selectOne(wrapper);
    if (provider == null) {
        throw new BusinessException("服务商不存在");
    }
    return provider;
}

/**
 * 查找服务商信息（允许不存在）
 * @param providerId 服务商ID
 * @param userId 用户ID
 * @return 服务商实体，可能为null
 */
public ProviderEntity findProvider(String providerId, String userId) {
    Wrapper<ProviderEntity> wrapper = Wrappers.<ProviderEntity>lambdaQuery()
        .eq(ProviderEntity::getId, providerId)
        .eq(ProviderEntity::getUserId, userId);
    return providerMapper.selectOne(wrapper);
}
```

### 3.2 更新操作
- 必须包含用户 ID 作为操作上下文。
- 采用直接更新方式，避免“先查询后更新”模式。
- 更新失败应抛出具体异常。
- 更新 SQL 必须具备明确 `where` 条件，禁止全表更新。
- 数据库表设计禁止使用物理外键，关联关系由业务逻辑维护。
- 对查询频繁字段必须建立索引并在增量 DDL 中记录。

## 4. 校验规范

### 4.1 三层校验机制
1. **Controller 层校验**
   - 基础数据格式校验。
   - 使用 `@Validated` 注解。
   - 校验字段格式、必填项、长度范围等。
2. **Service 层校验**
   - 业务规则校验。
   - 数据存在性检查。
   - 操作权限验证。
3. **Mapper 层约束**
   - 保证 SQL 条件完整性。
   - 保证写操作影响行数可检查。

## 5. 文档管理
- 所有 SQL 脚本放在 `docs/sql` 目录下。
- 设计文档放在 `docs/design` 目录下。
- API 文档基于 OpenAPI 3 规范生成并通过 Knife4j 展示，Apifox 可通过 OpenAPI 文档导入。
- 所有 Controller 接口必须标注文档注解（接口分组、接口说明、鉴权要求）。
- 所有接口入参对象和出参对象必须标注字段级文档注解，确保文档可直接用于联调。
- 后端项目任何架构变更、数据库表结构变更，都必须同步记录到后端技术文档与增量 DDL。
- 数据库结构变更只能新增 `Vx__*.sql` 增量脚本，禁止直接覆盖历史版本脚本。

## 6. 安全规范
- 所有涉及密钥相关数据必须加密存储，禁止明文存储。
- 所有用户输入都必须经过安全校验。
- 对外输出和日志信息必须进行敏感信息脱敏。

## 7. 开发规范

### 7.1 Entity JSON 字段处理
- Entity 涉及常量、JSON 时，必须使用对象进行包装。
- 表字段通过 JSON 进行映射。
- 通过 `org/xhy/infrastructure/converter` 进行转换。
- 通过 `org/xhy/infrastructure/config/MyBatisTypeHandlerConfig.java` 进行注册。

### 7.2 Entity 常量字段处理
- Entity 涉及常量时必须使用常量表示。
- 常量必须使用全大写命名，例如 `MCP`、`FUNCTION_CALLING`。

### 7.3 事务注意事项
- 必须在 Service 层编写事务。
- 禁止在 Mapper 层编写事务。

## 8. 日志规范
1. 日志打印使用中文。
2. 日志中不允许打印密钥，如需展示请使用 `***` 脱敏。

## 9. Git 规范
- 分支类型：
  - `master`：稳定版
  - `develop`：开发版（最新代码）
  - 功能分支：命名要求 `YYYY-MM-DD-feat/功能说明`
- 分支开发要点：
  1. 从 `master` checkout 创建功能分支进行开发。
  2. 不可从 `develop` checkout 创建功能分支。
  3. PR 需要同时提交到 `master` 和 `develop`。
