# 🏗️ 企业级知识库 Agent 系统 - 详细技术规格书 (Spec V2.0)

## 1. 项目概述

本项目旨在构建一个企业级的多文档知识库问答系统。项目采用**前后端分离**架构：
- **后端**：采用 **NestJS (MVC)** 作为底层框架，结合 **LangGraph.js** 进行复杂的多智能体与 Agentic RAG 状态机流式编排，并利用 **LlamaIndex.ts** 解决复杂的文档解析与三级分块（Auto-merging）问题。最终通过标准的 RESTful API 和 MCP 协议对外提供服务。
- **前端**：采用 **Next.js** + **React**，负责可视化交互与 SSE 流式数据解析展示。
- **包管理工具**：统一使用 **pnpm**。

***

## 2. 核心基础设施与技术栈选型

### 2.1 核心框架与库

- **包管理工具**：`pnpm` (全局统一)
- **后端服务**：NestJS (TypeScript, Node.js v20+)
- **ORM 与数据库**：Prisma ORM + PostgreSQL 15+ (本地原生安装，需配置 `pgvector` 扩展，**不使用 Docker**)
- **缓存与消息队列**：Redis + BullMQ (本地原生安装，**不使用 Docker**)
- **AI 对话编排**：LangGraph.js (`@langchain/langgraph`) + LangChain.js (`@langchain/core`, `@langchain/openai`)
- **RAG 数据处理**：LlamaIndex.ts (专用于文档解析、Node 树状层级管理)
- **外部协议集成**：`@modelcontextprotocol/sdk` (暴露 MCP Server)

### 2.2 项目初始化规范 (NestJS)

为了保证前后端分离结构和包管理的一致性，后端的初始化步骤如下：

```bash
# 使用 pnpm 调用 nest cli 初始化后端项目
pnpm dlx @nestjs/cli new backend --package-manager pnpm

# 进入后端目录
cd backend

# 安装核心依赖示例
pnpm add @prisma/client @nestjs/bull bullmq
pnpm add -D prisma
```

### 2.3 依赖的第三方 AI 模型 (API)

- **大语言模型 (LLM)**：OpenAI `gpt-4o-mini` (用于路由和改写) / `gpt-4o` (用于最终总结)。
- **向量模型 (Embedding)**：OpenAI `text-embedding-3-small` (输出 1536 维向量)。
- **重排序模型 (Reranker)**：Cohere `rerank-multilingual-v3.0` 或 硅基流动 BGE-Reranker。

## 3. NestJS 模块化设计 (Module Architecture)

系统严格按照领域驱动设计 (DDD) 划分为以下 5 个核心 Module。为了保持代码结构清晰，每个 Module 内部采用经典的三层/四层架构分包：
- **`controllers/`**: 负责处理 HTTP/SSE 请求，参数校验 (DTO)，返回响应。
- **`services/`**: 核心业务逻辑层。
- **`repositories/`** (类似 Mapper): 负责封装与 Prisma 的数据库交互逻辑。
- **`dto/`**: 数据传输对象定义。
- **`nodes/`** (仅在 RAG/Chat 模块): 用于定义 LangGraph 的核心节点逻辑。

### 3.1 `PrismaModule` (全局数据库模块)
- **职责**：作为全局模块，封装 PrismaClient，对外提供 `PrismaService` 供所有 Repository 调用，确保数据库连接的单例模式。

### 3.2 `DocumentModule` (文档与数据摄入)

- **目录结构**：
  ```text
  src/document/
  ├── controllers/
  │   └── document.controller.ts     # POST /api/documents/upload, GET /api/documents/:id/status
  ├── services/
  │   ├── document.service.ts        # 处理上传逻辑，写入 DB 并触发队列
  │   └── document-processor.processor.ts # BullMQ 消费者，执行 LlamaIndex 解析
  ├── repositories/
  │   ├── document.repository.ts     # Document 表 CRUD
  │   └── node.repository.ts         # L1/L2/L3 Node 表插入与级联管理
  └── dto/
      └── upload-document.dto.ts
  ```
- **职责**：处理用户上传文件，存入临时目录；将任务推入 BullMQ 队列；后台 Worker 执行 PDF/Excel 解析，调用 LlamaIndex 进行 L1/L2/L3 分块，最后将 L3 向量存入 `pgvector`。

### 3.3 `RAGModule` (核心检索服务)

- **目录结构**：
  ```text
  src/rag/
  ├── services/
  │   └── rag.service.ts             # 封装完整的 retrieve_pipeline
  ├── repositories/
  │   └── vector.repository.ts       # 执行 pgvector 混合检索 (Dense + Sparse)
  └── nodes/
      ├── retrieve.node.ts           # 初次/二次召回逻辑 + Auto-merging
      ├── grade.node.ts              # 相关性打分 LLM 调用
      └── rewrite.node.ts            # 查询重写 LLM 调用
  ```
- **职责**：封装企业级检索能力，对外暴露供 LangGraph 调用的 RAG Nodes。
- **检索流水线 (Pipeline)**：
  1. **初次召回 (`retrieve_initial`)**:
     - 仅对 `level == 3` (叶子块) 执行 Hybrid 检索（Dense + BM25 Sparse + RRF 融合）。
     - 获取较大候选集后，调用 Reranker 进行精排。
     - 对精排后的 L3 块执行 **Auto-merging**，若满足阈值则从数据库读取父块 (L3 -> L2 -> L1)。
  2. **相关性打分门控 (`grade_documents`)**:
     - 驱动轻量级 LLM 使用结构化输出对初次召回结果打分 (`yes/no`)。
  3. **查询重写路由 (`rewrite_question`)**:
     - 门控失败时，生成 `rewrite_query` 或 `hypothetical_doc` 等中间结果。
  4. **二次召回 (`retrieve_expanded`)**:
     - 使用重写后的查询再次执行步骤 1 的逻辑。

### 3.4 `ChatModule` (对话与流式响应)

- **目录结构**：
  ```text
  src/chat/
  ├── controllers/
  │   └── chat.controller.ts         # POST /api/chat/stream
  ├── services/
  │   ├── chat.service.ts            # 初始化 LangGraph 状态机，监听 streamEvents
  │   └── session.service.ts         # 历史会话管理
  ├── repositories/
  │   ├── session.repository.ts      # Session 表 CRUD
  │   └── message.repository.ts      # Message 表 CRUD
  └── graph/
      ├── agent.state.ts             # LangGraph State 接口定义
      └── agent.graph.ts             # 组装 Nodes 和 Edges 的路由逻辑
  ```
- **职责**：处理前端发来的聊天请求，管理 `Session` 和 `Message` 持久化，执行基于 LangGraph 的大堂经理 (Router) 逻辑，并向前端返回 SSE 数据流。

### 3.5 `MCPModule` (MCP 协议暴露)

- **目录结构**：
  ```text
  src/mcp/
  └── services/
      └── mcp-server.service.ts      # 生命周期 onModuleInit 时启动 STDIO/SSE Server
  ```
- **职责**：在 NestJS 生命周期 `onModuleInit` 时启动 MCP Server，将系统能力暴露给 Cursor/Trae 等外部工具。

## 4. 数据库设计详解 (Prisma Schema)

```prisma
generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["postgresqlExtensions"]
}

datasource db {
  provider   = "postgresql"
  url        = env("DATABASE_URL")
  extensions = [vector] // 必须启用 pgvector
}

// ================== 对话记忆模块 ==================
model User {
  id        String    @id @default(uuid())
  username  String    @unique
  sessions  Session[]
}

model Session {
  id        String    @id @default(uuid())
  userId    String
  user      User      @relation(fields: [userId], references: [id])
  title     String    @default("新会话")
  summary   String?   // 用于长对话的压缩摘要
  messages  Message[]
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
}

model Message {
  id        String    @id @default(uuid())
  sessionId String
  session   Session   @relation(fields: [sessionId], references: [id], onDelete: Cascade)
  role      String    // 'user' | 'assistant' | 'system' | 'tool'
  content   String    // 消息文本或序列化的 ToolCall JSON
  createdAt DateTime  @default(now())
}

// ================== 知识库 RAG 模块 ==================
model Document {
  id        String    @id @default(uuid())
  filename  String
  fileType  String    // 'pdf', 'csv', 'md' 等
  status    String    // 'PENDING', 'PROCESSING', 'COMPLETED', 'FAILED'
  error     String?   // 解析失败的原因
  nodes     Node[]    // 关联的所有层级节点
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
}

// 使用 Node 表统一存储 L1/L2/L3 多级结构 (支持 Auto-merging)
model Node {
  id         String    @id @default(uuid())
  documentId String
  document   Document  @relation(fields: [documentId], references: [id], onDelete: Cascade)
  
  level      Int       // 1: 根节点(大), 2: 中节点, 3: 叶子节点(小)
  parentId   String?   // 指向上一级 Node 的 ID
  
  content    String    // 切片后的纯文本内容
  metadata   Json?     // 存页码、标题等附加信息
  
  // 仅针对 L3 叶子节点生成向量，维度需与选用模型匹配 (如 OpenAI 为 1536)
  embedding  Unsupported("vector(1536)")? 

  @@index([documentId])
  @@index([parentId])
}
```

## 5. 核心 API 接口契约 (RESTful)

### 5.1 文档入库 (Document Ingestion)

**`POST /api/documents/upload`**

- **Content-Type**: `multipart/form-data`
- **Request**: `file` (文件二进制流)
- **Response**:
  ```json
  { "statusCode": 200, "data": { "documentId": "uuid", "status": "PENDING" } }
  ```
- **内部行为**：NestJS 接收文件，保存在本地/S3，创建 DB 记录，并将 `documentId` 推入 Redis BullMQ 队列。

**`GET /api/documents/:id/status`**

- **Response**:
  ```json
  { "statusCode": 200, "data": { "status": "COMPLETED", "progress": 100 } }
  ```

### 5.2 流式对话 (Chat Streaming) 与 可观测性数据推送

**`POST /api/chat/stream`**

- **Content-Type**: `application/json`
- **Request**:
  ```json
  {
    "sessionId": "uuid (可选，不传则新建)",
    "userId": "uuid",
    "message": "什么是公司2023年的年假政策？"
  }
  ```
- **Response**: `text/event-stream` (SSE 标准格式)
- **数据流结构 (基于 LangGraph streamEvents + 自定义 SSE 协议)**：
  前端可视化强依赖于流式过程中的实时状态推送。服务端在调用 LangGraph 的 `.streamEvents()` 时，捕获不同类型的事件（如 `on_chat_model_stream`, `on_tool_start`, `on_node_start`），将其转换为标准 SSE 格式下发。

  **SSE 流片段设计示例：**
  
  ```text
  event: text
  data: "根据《2023年员工手册》..."   // (大模型生成的 Token 块)
  
  event: tool_start
  data: {"toolName":"query_knowledge_base","args":{"query":"2023年假"}}
  
  event: trace
  data: {"step":"retrieve_initial","data":{"candidates":20,"top_scores":[0.92, 0.88]}}
  
  event: trace
  data: {"step":"grade_documents","data":{"decision":"no","reason":"未提及2023年"}}
  
  event: trace
  data: {"step":"rewrite_question","data":{"strategy":"hyde","new_query":"假设..."}}
  
  event: tool_end
  data: {"toolName":"query_knowledge_base","result":"[最终 RAG 文本]"}
  ```

## 6. 前端可视化与可观测规范 (Frontend Observability)

为了实现类似 Perplexity 或大厂内部知识库的“透明思考过程”，前端架构需实现以下交互：

### 6.1 思考过程折叠面板 (Thinking Process UI)

前端解析 SSE 流中的 `e:` (Custom Data) 帧，渲染一个动态更新的折叠面板：

1. **Tool 状态指示器**：当收到 `toolCall` 时，显示“🔍 正在查询知识库...”。
2. **检索漏斗可视化**：
   - 收到 `retrieve_initial`：显示“找到 20 篇相关片段，正在精排...”。
   - 收到 `grade_documents` 为 `no`：标红提示“片段相关性不足，正在启动查询重写”。
   - 收到 `rewrite_question`：展示改写后的新问题或 HyDE 伪答案（支持点击查看详情）。
   - 收到 `retrieve_expanded`：显示“二次召回完成，合并出 2 个大段落”。
3. **来源引用面板 (Citations)**：解析工具最终返回的 `result`，在聊天气泡下方展示引用的文档卡片（支持点击预览文档切片内容）。

### 6.2 异常处理策略 (Fallback)

1. **数据库宕机/Redis 断连**：NestJS 全局 Exception Filter 捕获，返回 `500 Internal Server Error`，前端降级提示。
2. **RAG 检索为空**：`search_knowledge_base` 工具必须返回 `{"status": "empty", "message": "知识库中未找到相关文档"}`，绝对禁止返回空字符串导致大模型幻觉。
3. **大模型 API 超时**：设置 15s 超时时间。超时后触发重试机制 (Retries = 2)。

### 6.2 链路追踪 (Tracing)

系统需在关键节点埋点，方便后期接入可视化平台：

- **LangSmith 集成**：由于我们采用了 LangGraph，可以直接无缝接入 LangSmith，实现对 Graph 运行轨迹（Trace）、Token 消耗量、各个 Node 耗时的全自动端到端可观测。
- 记录每次 RAG 的输入 (原始 Query) 与输出 (改写后的 Query)。
- 记录 Reranker 淘汰掉的 Document ID。
