# 前端异常监控体系：rrweb 接入与实现技术文档

## 1. 背景与目标

为了提升前端异常排查效率，解决“线上报错难以复现”、“用户操作路径不明确”的痛点，本项目引入 `rrweb` (Record and Replay the Web) 技术，构建一套轻量、高效的**异常触发式用户操作录屏与回放监控系统**。

**核心目标**：

- **无侵入性**：在不影响业务代码和主线程性能的前提下静默运行。
- **追溯式录制**：仅在发生异常时，截取并上报报错前 N 分钟的录像数据，极大节省带宽和存储成本。
- **高保真回放**：确保管理后台能够利用收集到的数据，完美还原异常发生时的 DOM 结构与用户交互。

***

## 2. 核心原理解析

`rrweb` 并非录制真实的 MP4 视频，而是基于 DOM 状态的快照与增量更新机制：

1. **全量快照 (Full Snapshot)**：序列化当前页面的完整 DOM 树，生成基础数据结构。
2. **增量监听 (Incremental Diff)**：利用 `MutationObserver` 监听后续的 DOM 增删改，拦截鼠标轨迹、点击、输入等交互事件。
3. **沙箱回放 (Replay)**：后台管理系统将基础 DOM 树与增量事件按时间轴组合，在 `iframe` 沙箱中重绘页面，实现“类似视频”的播放效果。

***

## 3. 前端 SDK 核心架构设计

前端 SDK 主要包含三大模块：**行为录制模块**、**内存管理模块**、**异常捕获与上报模块**。

### 3.1 行为录制与全量快照策略 (Checkpoint)

为了保证截取到的录像片段能够独立回放，必须确保截取的数据**以全量快照（Type: 2）开头**。

- **配置** **`checkoutEveryNms`**：强制 `rrweb` 定期重新生成一个全新的全量快照，作为后续增量事件的基准点。为了兼顾 CPU 性能与回放截断的颗粒度，建议默认设置为 30 秒（可配置）。
- **隐私脱敏**：默认开启 `maskAllInputs: true`，并通过特定 CSS 类名（如 `.rr-mask`）在序列化阶段跳过敏感信息（如密码、身份证号）。

### 3.2 内存滑动窗口机制 (Sliding Window)

C. 增加“双重阈值”兜底（高级设计）
在极端情况下（比如代码写了死循环疯狂修改 DOM），短短 10 秒内可能会产生上万个 rrweb 事件，直接撑爆内存。
因此， 优秀的架构不仅要看时间，还要看数量 。

合理的清理逻辑设计应该是：

```
const TIME_WINDOW = 3 * 60 * 1000; // 3分钟时间窗口
const MAX_EVENT_COUNT = 15000;     // 兜底数量窗口

function pruneBuffer() {
  // 1. 如果数量超过极值，强制丢弃最老的一半数据（牺牲回放，保页面不崩）
  if (events.length > MAX_EVENT_COUNT) {
     // 执行紧急降级清理...
     return;
  }

  // 2. 正常的基于“时间窗口”和“锚点快照”的清理逻辑
  const targetTime = Date.now() - TIME_WINDOW;
  let anchorIndex = -1;
  for (let i = events.length - 1; i >= 0; i--) {
     if (events[i].timestamp <= targetTime && events[i].type === 
     2) {
         anchorIndex = i;
         break;
     }
  }
  if (anchorIndex > 0) {
      events.splice(0, anchorIndex);
  }
}

// 启动高频平滑清理
setInterval(pruneBuffer, 10000);
```

### 总结

- **事件入队**：监听 `rrweb.record` 的 `emit` 回调，将所有产生的事件存入数组。
- **定期清理 (Prune)**：
  - 启动定时任务（如默认每 10 秒执行一次，可配置），清理过期数据。这种高频小批量的清理方式可以平滑 V8 引擎的垃圾回收（GC）压力，避免页面卡顿。
  - **清理规则（极其重要：锚点快照策略）**：不能简单按时间截断。正确逻辑是：计算时间窗口的起始点（如 `T_target = Date.now() - 3分钟`），在缓冲区中**倒序寻找最后一个在** **`T_target`** **之前（或刚好等于）的全量快照**（称之为锚点快照）。
  - **安全删除**：找到锚点快照后，仅删除该锚点快照**之前**的所有事件，保留锚点快照及其之后的所有数据。
  - **保证回放完整性**：此规则确保了截取的数据不仅**永远以全量快照开头**，而且**一定能完整覆盖甚至略微超出设定的 3 分钟窗口**，不会丢失窗口起始阶段的增量操作。

### 3.3 异常触发与高可靠上报 (Exception Trigger)

当发生 JS 运行时错误、Promise 异常或特定业务报错时：

1. **截取现场**：立即从内存滑动窗口中提取当前事件数组（即报错前约 3 分钟的操作记录）。
2. **数据压缩 (Web Worker)**：由于录像 JSON 数据庞大，`JSON.stringify` 和压缩操作会阻塞主线程导致严重卡顿。**必须**将事件数组发送至 Web Worker，在后台线程使用 `pako` 或 `fflate` 进行 Gzip 压缩。
3. **上报策略**：
   - **常规异常上报**：使用带 `keepalive: true` 标志的 `fetch` 请求，将压缩后的二进制流上传。确保即使页面崩溃或刷新，请求也能在后台发出。
   - **兜底离场上报**：监听 `visibilitychange` 事件，在页面隐藏/卸载前，使用 `navigator.sendBeacon` 发送最后残留的轻量级数据。

## 4. 数据流转时序图

```text
[用户操作页面] -> [MutationObserver 产生事件]
      |
      v
[检查是否到达 Checkpoint 时间 (如30s)] -> 是 -> [生成 FullSnapshot (Type: 2)]
                                     -> 否 -> [生成 Incremental (Type: 3)]
      |
      v
[存入内存 Buffer 数组]
      |
      v
[定时清理任务 (每10s)] -> [倒序寻找在时间窗口起点前的最后一个 Type:2 快照 (锚点快照)] -> [安全删除该锚点快照之前的事件]
      |
      v
[🔥 触发全局异常 (window.onerror)]
      |
      v
[提取当前 Buffer 数组 (确保以 Type:2 开头)]
      |
      v
[发送至 Web Worker 进行 GZIP 压缩，避免 UI 卡顿]
      |
      v
[使用 fetch(url, { keepalive: true }) 异步上报至后端]
```

## 5. 关键技术难点与解决方案总结

| 技术难点              | 产生原因                                                | 解决方案                                                                                                                                                              |
| :---------------- | :-------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **内存与带宽浪费**       | 全量持续录制并上传会导致极高的存储和流量成本。                             | 采用**滑动窗口缓冲 + 异常触发截取**策略，只上传报错前 3 分钟的有效现场。                                                                                                                         |
| **截取的数据无法回放**     | 直接按时间截断数组，首个事件通常是增量事件，缺少基础 DOM 树导致播放器白屏。            | 引入 **Checkpoint 机制** (`checkoutEveryNms`)，清理内存时采用**锚点快照策略**：倒序寻找时间窗口起点前最近的一个**全量快照**作为截取起点，确保完整覆盖窗口且以快照开头。                                                        |
| **打包上报导致页面卡顿**    | `JSON.stringify` 和 Gzip 压缩巨量对象是 CPU 密集型同步操作，会阻塞主线程。 | 将序列化和数据压缩逻辑**前置到 Web Worker** 中执行，实现主线程 0 阻塞。注意：Worker 通信需引入 RequestID 机制以支持并发上报。                                                                                 |
| **回放白屏且仅有鼠标轨迹**   | 播放器缺少 `Meta` 事件（Type: 4），无法获取页面初始尺寸导致容器塌陷。          | 在前端 SDK 中**永远缓存最近一次产生的** **`Meta`** **事件**。在截取数据并上传前，强行检查数据开头是否为 `Meta` 事件，如果不是，则将其插入数组最前面（`unshift`）。**正确的数据流开头必须是：\[Meta(Type:4), FullSnapshot(Type:2), ...]**。 |
| **报错后页面崩溃导致上报失败** | 传统的 XHR/axios 请求在页面刷新或关闭时会被浏览器强制取消。                 | 核心数据上报采用 **`fetch`** **+** **`keepalive: true`**；离场兜底采用 **`navigator.sendBeacon`**。                                                                               |
| **敏感数据泄露**        | 录制过程可能抓取到用户的密码、个人隐私信息。                              | 序列化时配置全局 `maskAllInputs` 屏蔽输入框，业务侧配合使用特定的忽略类名进行脱敏。                                                                                                                |

## 6. 核心 Bug 深度复盘：并发报错导致上传失败

在最初的架构实现中，我们遇到了一个经典的并发状态丢失问题：**“首次触发异常能成功上报录像，但短时间内再次触发异常会导致后续上报全部失败或数据错乱。”**

### 6.1 问题拆解与分析过程

**现象还原**：
当用户疯狂点击报错按钮（或页面短时间内触发多个 Promise 异常）时，发现 Network 面板中只有第一个请求发出了，或者后续请求带的 `errorInfo` 发生了张冠李戴。

**定位思路**：

1. **排查录制层**：检查 `rrweb` 的 buffer 是否因为第一次截取而被清空？—— 否，我们用的是 `slice` 深拷贝，原 buffer 数据完好。
2. **排查网络层**：是否因为 `keepalive` 请求并发过多被浏览器拦截？—— 检查发现，代码甚至没有走到 `fetch` 发送请求的那一步。
3. **排查异步通信层 (Web Worker)**：问题锁定在主线程与 Worker 的交互上。

**根本原因：回调被覆盖 (Race Condition)**

在最初的代码中，我们在每次触发异常时动态绑定 Worker 的回调：

```javascript
// 错误示例：动态绑定 onmessage
handleError(errorInfo) {
  this.worker.postMessage(uploadEvents);
  this.worker.onmessage = (e) => { 
    this.uploadData(e.data.data, errorInfo); // 依赖闭包中的 errorInfo
  };
}
```

因为 `this.worker` 是一个单例：

1. 触发报错 A：绑定回调 A（持有 `errorInfo A`），Worker 开始压缩 A。
2. 触发报错 B：在 A 压缩完之前，**覆盖绑定**为回调 B（持有 `errorInfo B`）。
3. Worker 完成 A 的压缩：返回结果，触发了**回调 B**！导致把 A 的录像配上 B 的报错信息发给了后端。随后 Worker 完成 B 时，逻辑彻底陷入混乱。

### 6.2 解决方案：引入 Request ID 机制

要解决这个异步通信的状态丢失问题，不能依赖闭包和动态绑定，必须建立\*\*“请求-响应”匹配机制\*\*。

**改造步骤**：

1. **持久化监听**：在 SDK 初始化（`constructor`）时，**仅绑定一次** `this.worker.onmessage`。
2. **任务状态管理**：引入 `this.pendingTasks = new Map()`，用于在主线程存储所有“正在压缩中”的上下文上下文。
3. **Request ID 匹配**：
   - 主线程每次发消息前，生成一个唯一的 `requestId`，并将 `errorInfo` 存入 Map：`pendingTasks.set(requestId, errorInfo)`。
   - 主线程将 `{ requestId, events }` 发送给 Worker。
   - Worker 压缩完毕后，将 `{ requestId, compressedData }` 原样返回。
   - 主线程在统一的 `onmessage` 中，通过 `requestId` 去 Map 中取回对应的 `errorInfo`，执行上传，然后从 Map 中删除该任务。

这种基于 `Request ID` 的设计模式，是前端处理高并发 Web Worker、Iframe postMessage 或 WebSocket 通信时的**标准架构级解决方案**，完美保证了数据的幂等性和状态隔离。

## 7. 性能优化与量化指标 (面试核心考点)

针对本项目的核心架构，我们实施了多项关键优化。以下是具体的优化点、量化指标以及如何验证优化效果的说明：

### 7.1 内存滑动窗口优化 (O(1) 空间复杂度)

- **优化背景**：`rrweb` 默认会持续记录所有 DOM 变化，如果用户在页面停留 1 小时，事件数组会无限增长，最终导致浏览器 OOM (Out of Memory) 崩溃。
- **量化指标**：
  - **内存占用上限**：通过配置 `recordWindow: 3min` 和 `maxEventCount: 10000`，使 SDK 的内存占用在达到阈值后进入动态平衡（通常维持在 `10MB - 30MB` 之间），不再随时间线性增长。
  - **GC 耗时**：通过 `10s` 一次的高频平滑清理，将垃圾回收（GC）分散，避免长达 `100ms+` 的 Full GC 导致页面卡顿。
- **验证手段**：
  - Performance Monitor：打开 Chrome DevTools，观察 `JS Heap Size`。模拟用户操作 30 分钟，确认堆内存曲线呈“锯齿状”且波峰保持在稳定水平。
  - **Heap Snapshot**：在运行 5 分钟和 50 分钟时分别拍一次内存快照，对比 `rrweb` 事件数组的 `Retained Size`，确认其没有显著膨胀。

### 7.2 Web Worker 零阻塞压缩优化

- **优化背景**：当发生报错时，我们需要将庞大的 JSON 事件数组（可能达数 MB）进行 `JSON.stringify` 和 `Gzip` 压缩。如果在主线程同步执行，会导致页面瞬间卡死（Jank）。
- **量化指标**：
  - **主线程阻塞时间 (TBT)**：优化前，处理 5MB 的 JSON 可能会产生 `200ms - 500ms` 的 Long Task（长任务）；优化后，主线程耗时降至 `< 16ms`（仅计算 `postMessage` 的开销），实现真正的 **0 阻塞**。
  - **数据压缩率**：使用 `fflate` 的 Gzip 压缩，录像数据的体积可减少 **80% - 90%**（例如 5MB 压缩至 500KB 左右），极大降低了网络 I/O 和后端存储压力。
- **验证手段**：
  - **Performance 面板**：在点击“触发报错”瞬间录制性能轨迹，观察 `Main` 线程。优化前会看到大段的红色 `Long Task`；优化后，长耗时计算转移到了 `Worker` 线程轨道。
  - **Network 面板**：查看 `upload` 请求的 Payload，确认 `Content-Encoding: gzip` 且传输体积符合预期的压缩比。

### 7.3 Request ID 并发状态隔离优化

- **优化背景**：当页面短时间内抛出多个异常（如循环报错）时，由于 Worker 异步通信的特性，会导致后续报错的回调覆盖前一个回调，出现“A 录像配了 B 错误信息”的串联 Bug。
- **量化指标**：
  - **并发匹配准确率**：在极高并发下（如 `1s` 内触发 10 次报错），保证 Worker 返回的压缩数据与报错元数据（Message, Stack）的匹配准确率达到 **100%**。
- **验证手段**：
  - **自动化并发测试**：编写脚本在 `100ms` 内连续触发 5 次不同的报错。
  - **数据一致性校验**：前往后端管理系统，检查生成的 5 条录像回放记录。确认每条记录的堆栈信息与其对应的用户操作画面完美契合，且 Network 没有丢失任何一次上报请求。

## 8. 后续接入指南

开发人员接入本 SDK 后，只需在应用入口处进行初始化，即可实现全自动的异常录制与上报。SDK 提供了灵活的配置项，支持自定义录制窗口和清理频率，以适应不同复杂度的业务场景：

```javascript
import ExceptionMonitor from '@company/monitor-sdk';

ExceptionMonitor.init({
  dsn: 'https://api.monitor.example.com/upload', // 上报接口
  
  // --- 录制与内存策略配置 ---
  recordWindow: 3 * 60 * 1000, // 录制窗口时间：保留报错前 3 分钟的录像（默认 3分钟）
  checkoutEveryNms: 30 * 1000, // 全量快照频率：每 30 秒生成一次全量快照（默认 30秒）
  pruneInterval: 10 * 1000,    // 清理任务频率：每 10 秒执行一次内存清理（默认 10秒）
  maxEventCount: 10000,        // 兜底策略：缓冲区最大事件数量，防止极端情况内存溢出
  
  // --- 隐私与安全配置 ---
  maskAllInputs: true, // 默认屏蔽所有输入框内容
  ignoreClass: 'rr-mask' // 业务侧敏感 DOM 标识类名
});
```

**配置建议**：

- **普通页面**：使用默认配置即可（3分钟窗口 / 30s快照 / 10s清理）。
- **复杂重型页面（如低代码平台、大型表格）**：为避免生成快照时阻塞主线程，建议调大 `checkoutEveryNms`（如 `60 * 1000`）。
- **轻量级 C 端营销页**：为了更精准的录像切片，可以调小快照频率（如 `15 * 1000`）。
- **黄金法则**：`pruneInterval`（清理频率）必须小于 `checkoutEveryNms`（快照频率），建议 `pruneInterval = checkoutEveryNms / 3`。
